# ADBPG 长期记忆 — 电商 SaaS 模式

面向多商家 SaaS 平台（电商 ERP、聚水潭类系统），提供商家/用户两级记忆隔离，支持商家公共知识与用户私有记忆的优先级合并。

## 方案概览

```
┌─────────────────────────────────────────────────┐
│  SaaS 平台                                       │
│                                                   │
│  商家 A（Agent A）          商家 B（Agent B）      │
│  ├── 公共知识               ├── 公共知识           │
│  │   退货政策、产品目录       │   退货政策、产品目录  │
│  │                          │                     │
│  ├── 用户张三（私有记忆）    ├── 用户王五（私有记忆） │
│  │   偏好、特殊条款          │   偏好、特殊条款      │
│  │                          │                     │
│  └── 用户李四（私有记忆）    └── 用户赵六（私有记忆） │
│      偏好、特殊条款              偏好、特殊条款      │
└─────────────────────────────────────────────────┘
                        │
                        ▼
┌─────────────────────────────────────────────────┐
│  ADBPG Memory Service                            │
│  所有记忆统一存储，通过 metadata 过滤实现隔离       │
└─────────────────────────────────────────────────┘
```

核心原则：
- 商家之间完全隔离，互不可见
- 同一商家下，用户之间私有记忆隔离
- 商家公共知识所有用户可见
- 用户私有记忆优先于商家公共知识
- 所有 session 共享到用户级别，不做 session 隔离

## 隔离模型

通过 adbpg-mem CLI 的三个 scope 参数实现：

| SaaS 层级 | 参数 | 示例 | 说明 |
|-----------|------|------|------|
| 商家 | `-a <商家ID>` | `-a merchant_001` | 商家间完全隔离 |
| 用户 | `-u <用户ID>` | `-u zhangsan` | 同一商家下用户间隔离 |
| 会话 | 不使用 `-r` | — | 所有 session 共享到用户 |

存储示例：

```bash
# 商家公共知识（所有用户可见）
adbpg-mem add "退货期限7天" -a merchant_001

# 用户私有记忆（仅该用户可见）
adbpg-mem add "张三的折扣是85折" -a merchant_001 -u zhangsan
```

检索示例：

```bash
# 搜用户私有：只搜到张三自己的
adbpg-mem search "折扣" -a merchant_001 -u zhangsan

# 搜商家公共：搜到所有不带 -u 存储的记忆
adbpg-mem search "退货" -a merchant_001
```

## 部署指南

### 1. 安装 adbpg-mem CLI

```bash
cd adbpg-memory-cli/node
npm install && npm link
```

### 2. 配置数据库连接

```bash
adbpg-mem init
# 选择 REST 或 SQL 模式，填入连接信息
```

所有商家共享同一个数据库连接，隔离通过 metadata 过滤实现。

### 3. 为每个商家部署 Agent

每个商家对应一个 CoPaw Agent，使用电商版 SKILL.md：

```bash
AGENT_SKILLS=~/.copaw/workspaces/<agent_id>/skills
mkdir -p "$AGENT_SKILLS/adbpg_memory"
cp examples/saas-ecommerce/SKILL.md "$AGENT_SKILLS/adbpg_memory/SKILL.md"
```

### 4. 配置商家 ID

在 Agent 的 PROFILE.md 中记录商家 ID：

```markdown
## 记忆配置
- **商家ID：** merchant_001
- **记忆模式：** 两级隔离（商家公共 + 用户私有）
```

### 5. 启用技能

```bash
curl -X POST http://127.0.0.1:8088/api/skills/adbpg_memory/enable \
  -H "X-Agent-Id: <agent_id>"
```

## 商家公共知识录入

### 对话录入（推荐日常使用）

管理员直接和 Agent 对话：

```
管理员：记录商家知识：退货政策是7天无理由，运费由买家承担
管理员：记录商家知识：VIP客户退货期限延长到15天
管理员：记录商家知识：SKU-001 蓝色T恤 售价99元 库存500件
```

Agent 识别到"记录商家知识"，存储时只带 `-a` 不带 `-u`。

### 文件批量导入（推荐初始化）

将商家知识整理为文本文件，逐条导入：

```bash
# merchant_knowledge.txt（一行一条知识）
退货政策是7天无理由，运费由买家承担
VIP客户退货期限延长到15天
SKU-001 蓝色T恤 售价99元 库存500件
SKU-002 白色衬衫 售价159元 库存300件

# 批量导入
while IFS= read -r line; do
  adbpg-mem add "$line" -a merchant_001 --agent
  sleep 2
done < merchant_knowledge.txt
```

### 对话触发文件导入

管理员上传文件，让 Agent 处理：

```
管理员：导入商家公共知识，文件在 /tmp/product_catalog.txt
```

### ERP 系统自动同步

通过脚本定期从 ERP 同步：

```bash
#!/bin/bash
# 从 ERP API 导出产品信息并同步到记忆
curl -s https://erp.example.com/api/products \
  | jq -r '.[] | .name + " 售价" + .price + "元 库存" + .stock + "件"' \
  | while read line; do
      adbpg-mem add "$line" -a merchant_001 --agent
      sleep 2
    done
```

## 使用长期记忆

启用后，用户无需学习任何命令，正常对话即可。

### 自动触发

当用户提到个人偏好、做出决策、分享重要信息时，Agent 自动存储为用户私有记忆。

### 主动触发

```
用户：记住我偏好顺丰发货
用户：我之前的退货期限是多少？
用户：你知道关于我的哪些信息？
```

### 跨会话记忆

所有 session 的记忆共享到用户级别。用户换个对话窗口，Agent 仍然记得之前的信息。

## 记忆优先级与合并

Agent 每次回答前执行两次搜索：

```
第一次：用户私有记忆（-a merchant_001 -u zhangsan）
第二次：商家公共记忆（-a merchant_001）
```

合并规则：

| 场景 | 处理方式 | 示例 |
|------|---------|------|
| 同一主题有冲突 | 用户私有优先 | 公共"退货7天" vs 私有"退货15天" → 采用15天 |
| 用户私有没有 | 用商家公共补充 | 用户无物流偏好 → 用公共"支持顺丰和圆通" |
| 两者都有且一致 | 去重，只展示一次 | 都说"支持顺丰" → 展示一次 |

回复中标注来源：

```
根据记忆：
- [私有] 您的退货期限是15天（特殊条款）
- [公共] 支持顺丰和圆通发货
```

## 对接指南

### 商家 ID 的传入

每个商家对应一个 CoPaw Agent，商家 ID 通过以下方式确定：

1. Agent 的 `agent_id` 直接作为商家 ID
2. 或在 PROFILE.md 中配置映射关系

### 用户 ID 的传入

用户 ID 由接入渠道自动传入：

| 渠道 | 用户 ID 来源 |
|------|-------------|
| DingTalk | 钉钉用户 ID |
| 企业微信 | 企微用户 ID |
| Discord | Discord 用户 ID |
| Console | 手动指定或默认 |

CoPaw 的渠道层会将用户 ID 传递给 Agent，Agent 在调用 adbpg-mem 时带上 `-u`。

### 角色区分

| 角色 | 能做什么 | 不能做什么 |
|------|---------|-----------|
| 管理员 | 录入/修改商家公共知识 | 查看其他商家的记忆 |
| 普通用户 | 存取自己的私有记忆，查看商家公共知识 | 查看其他用户的私有记忆 |

角色区分通过渠道层的权限控制实现，不在记忆层面处理。

### 与现有系统集成

```
ERP/OMS 系统
    │
    ├── 产品目录变更 → 脚本调用 adbpg-mem add -a merchant_001
    ├── 政策更新 → 管理员对话录入
    └── 用户标签同步 → 脚本调用 adbpg-mem add -a merchant_001 -u <user_id>
```

## 常见问题

### 用户搜到了其他用户的记忆

检查存储时是否正确带了 `-u`。不带 `-u` 的记忆是商家公共知识，所有用户都能搜到。这是设计如此，不是 bug。

### 用户搜不到商家公共知识

确认 Agent 是否执行了第二次搜索（只带 `-a`）。如果只做了一次搜索（带 `-a` + `-u`），就搜不到公共知识。

### 商家 A 的用户搜到了商家 B 的记忆

检查 `-a` 参数是否正确。每次调用必须带正确的商家 ID。

### 用户私有记忆没有覆盖公共知识

检查 Agent 是否按优先级合并了两次搜索结果。SKILL.md 的 description 中有明确的合并规则。

### 批量导入很慢

每条记忆需要 LLM 提取事实 + Embedding 向量化，建议每条间隔 2 秒。大批量导入（>1000 条）建议分批执行。

### 如何更新商家公共知识

目前 CLI 不支持单条更新。可以存入新版本的知识，旧版本会因为语义相似度较低而在搜索中自然排到后面。后续版本将支持 update 和 delete 操作。
